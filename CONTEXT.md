# CONTEXT.md — 可复用关键上下文（压缩版）

> 目的：把“题目要求 + 产品方向 + 关键约束 + 架构要点”压缩成一份可直接复用/喂给模型的上下文包。

## 1) 一句话目标

做一个「慢病用药小管家」网页应用，帮助用户建立个人用药计划、按时提醒与复诊/续方提示，并展示 AI 在医疗场景的边界与成本意识。

## 2) 题目要点（面试官想看到的）

- 从“问诊→购药→用药→续方”的链路里，选一个切入点做闭环（本项目聚焦：用药 + 续方）
- 体现 AI PM 的关键素质：
  - 用户痛点与场景拆解（老年人、慢病依从性）
  - AI 能力边界（不诊断、不替代医嘱）
  - 成本意识（token 分层，能不用 AI 就不用）
  - 人机协作（AI 提供参考，人做决定）

## 3) 核心用户故事（MVP）

- 作为慢病患者，我想把每日要吃的药按时间段排好，**点击即可打卡**，不再漏服/重复吃。
- 作为患者/家属，我想一键开启**关怀模式**（大字、大按钮），老人也能用。
- 作为患者，我想在新增药物时快速查一下与现有用药是否冲突（**AI 相互作用查询**）。
- 作为患者，我想看到血糖/血压/心率（哪怕是 mock）与用药管理联动，证明产品不是孤岛。
- 作为患者，我想在处方到期前收到提示，及时线上续方。

## 4) 功能范围（本 repo 已实现）

- **今日用药**：时间线（早/中/晚/睡前）+ 一键打卡 + 进度条依从率
- **药箱管理**：病种分类（糖尿病/高血压/高血脂）+ 常见药物模板 + 自定义新增/删除
- **药物相互作用查询**：BYOK 调用 OpenAI-compatible API；无 Key 时走 mock 结果
- **健康数据（设备 mock）**：血糖仪/血压计/手表 7 天数据；血糖偏高时给出“建议关注”提示
- **续方管理**：处方到期倒计时（红/黄/绿）+ “去续方”按钮（占位）
- **关怀模式**：一键切换（CSS 缩放 rem，避免两套 UI）

## 5) 不做什么（刻意取舍）

- 登录/账号体系：MVP 无收益，增加复杂度
- 真实设备 SDK：复杂度高，用 mock 证明思路
- 推送通知：涉及权限/Service Worker，超出 demo 范围
- 医疗诊断/治疗建议：严格不做

## 6) AI 风险边界（必须持续出现）

- AI **不诊断、不开处方、不替代医嘱**
- AI 输出统一强调：**仅供参考，需咨询医生/药师**
- 健康数据异常提示使用“建议关注/如持续请就医”，避免诊断性措辞

## 7) Token/模型策略（省钱优先）

- 0 token：时间线提醒、续方倒计时、设备异常阈值提示（本地规则）
- 低成本模型：药物相互作用查询默认 `gpt-4o-mini`（约 200 token/次）
- 仅在必要时才上更贵模型（例如：复杂多药联用、特殊人群用药风险汇总；本 demo 未启用）

## 8) 代码结构（快速定位）

```
src/
├── app/
│   ├── api/check-interaction/route.ts  → AI 相互作用接口
│   └── page.tsx                        → 单页入口（底部 tab 切换）
├── components/                         → 5 个页面视图 + header/nav
└── lib/
    ├── types.ts                        → 核心类型
    ├── mock-data.ts                    → 模板/相互作用 mock/设备数据生成
    ├── store.ts                        → localStorage 持久化
    └── context.tsx                     → AppProvider 全局状态
```

## 9) AI 接口约束（route.ts 的最小承诺）

- 输入：`drug1`, `drug2`, `key`, `baseUrl`, `model`
- 输出：若无相互作用 → `{ interaction: null }`
- 有相互作用 → `{ interaction: { drug1, drug2, severity, description, recommendation } }`
- 强制 JSON 输出（`response_format: { type: 'json_object' }`），避免解析成本与模型跑偏
