# DEVELOPMENT.md — 慢病用药小管家 · 设计决策记录

## 1. 产品定位

> **不是医疗 App，是用药管理助手。**

核心链路：问诊 → 购药 → **用药（本产品聚焦）** → 续方

目标用户：慢性病患者（糖尿病、高血压、高血脂），多为中老年人。

## 2. 关键设计决策

| 决策 | 原因 |
|---|---|
| **病种驱动药物模板** | 不同慢病用药模式差异大（胰岛素跟餐绑定 vs 降压药定时），模板降低老年用户输入门槛 |
| **关怀模式（适老化）** | 目标用户多为老年人，一键切换放大字号+间距，用 CSS 变量缩放 rem，不做两套 UI |
| **设备数据 Mock 接口** | 展示系统级思维——产品不是孤岛，血糖仪/血压计/手表数据可触发智能提醒。真实对接复杂度高，用 mock 证明设计意图 |
| **AI 风险边界：行内提示** | 每个 AI 输出页面底部附加医疗免责声明，不做重型系统，但处处可见"仅供参考，遵医嘱" |
| **Token 消耗分层** | 药物相互作用查询用 gpt-4o-mini（成本低、医学知识够用）；提醒文案、格式化用本地规则零 token 消耗 |
| **BYOK + Mock 双模式** | 面试官不需要 API Key 就能完整体验，配了 Key 后 AI 功能无缝启用 |
| **续方倒计时** | 直接回应题目"问诊→购药→用药→续方"的完整链路，处方到期前自动变红 |
| **药物相互作用检查** | 最能体现"AI 在医疗场景正确使用方式"的功能——AI 辅助判断，人做最终决策 |

## 3. Token 消耗策略

```
┌──────────────────────┬──────────────────┬────────────┐
│ 功能                  │ 执行方式          │ Token 消耗  │
├──────────────────────┼──────────────────┼────────────┤
│ 用药提醒 / 时间线      │ 本地规则           │ 0          │
│ 续方倒计时             │ 日期计算           │ 0          │
│ 设备数据异常提示        │ 阈值判断           │ 0          │
│ 药物相互作用查询        │ AI (gpt-4o-mini)  │ ~200 token │
│ 复杂药理分析（预留）    │ AI (gpt-4o)       │ ~800 token │
└──────────────────────┴──────────────────┴────────────┘
```

设计原则：能用规则解决的不调 AI，调 AI 时用最便宜够用的模型。

## 4. AI 医疗风险边界

- AI **不诊断、不开处方、不替代医嘱**
- 每个 AI 输出页面底部均有 `⚕️ 仅供参考` 免责提示
- 药物相互作用结果始终附带"请咨询医生或药师"
- 设备数据异常提示措辞为"建议关注""如持续偏高请及时就医"，不做诊断

## 5. 技术选型

| 层 | 技术 | 原因 |
|---|---|---|
| 框架 | Next.js 16 (App Router) | SSR/SSG + API Routes 一体化 |
| 语言 | TypeScript (strict) | 类型安全 |
| 样式 | Tailwind CSS v4 | 原子化 CSS，关怀模式用 CSS 变量缩放 |
| 状态 | React Context + localStorage | 足够简单，无需引入状态库 |
| AI | OpenAI-compatible API | BYOK 模式，兼容 OpenAI/DeepSeek/其他 |

## 6. 未做的事（及原因）

| 未做 | 原因 |
|---|---|
| 用户登录/注册 | MVP 不需要，增加复杂度无收益 |
| 真实设备 SDK 对接 | 复杂度极高，mock 足以证明设计意图 |
| 推送通知 | 需要 Service Worker + 权限，超出 demo 范围 |
| 语音提醒 | 改变产品重心，属于不同产品方向 |
| 多患者管理 | 单用户场景足够展示核心价值 |
